# Z-MOS Gen 4 Requirements Document (RD)

- Document Code: `RD-GEN4-001`
- Version: `0.1`
- Status: Draft
- Baseline: Z-MOS v3.0.1 "Purity"

---

## 1. Vision & Objectives

### 1.1 Proposed Codename

- **Z-MOS v4.0 "Ascension"**
- Alternate candidates: **Sovereignty**, **Harmony**, **Nexus**

### 1.2 Vision Statement

Z-MOS Gen 4 advances the Purity baseline from a deterministic, single-node enforcement engine into a distributed, intelligence-aware sovereign runtime. Gen 4 preserves the Pure Truth-First foundation while introducing intelligent adaptive governance, multi-agent coordination, and a living truth model that can evolve safely without compromising canonical authority.

### 1.3 Objectives

1. **Preserve canonical authority**
   - Retain `.z-mos/truth.contract.json` as the authoritative runtime truth source.
   - Retain `.z-mos/intent.card.json` as the canonical execution intent boundary.

2. **Enable adaptive governance**
   - Move beyond static hard-block rules to governance that can adapt to runtime evidence and authorized evolutionary changes.

3. **Support distributed sovereignty**
   - Extend the current single-workspace model to support federated/multi-node execution and truth synchronization.

4. **Increase intelligence without sacrificing safety**
   - Add meta-reasoning capabilities, while preserving fail-closed safety for any unknown or unauthorized state.

5. **Maintain auditability and trace integrity**
   - Keep append-only trace structure and evidence-based verification as a first-class requirement.

6. **Reduce operational friction**
   - Improve the upgrade, bootstrap, and recovery experience so that Gen 4 is easier to adopt without weakening trust assumptions.

## 2. Core Principles

These principles are non-negotiable inheritances from Gen 3 Purity. They must be preserved in any Gen 4 design.

### 2.1 Pure Truth-First Authority

- All governance decisions are resolved from canonical authority files.
- The engine must not depend on fallback session artifacts or legacy state for enforcement.
- The current `.z-mos/state/runtime-state.json` is explicitly helper/advisory only.

### 2.2 Hard-Block Safety Baseline

- Hard-block enforcement remains the default mode.
- Unknown state, missing evidence, or authority conflicts must result in fail-closed behavior.
- Warnings may exist, but they must never bypass hard-block conditions.

### 2.3 Explicit Execution Intent

- Execution must always be bounded by an explicit, human-authorized intent card.
- Intent scope, allowed tools, host/database access, and termination conditions must be declared and enforced.

### 2.4 Immutable Audit & Verification

- Execution traces must remain append-only and verifiable.
- The trace layer is a trust foundation, not a convenience log.
- Verification evidence should be integral to operations like `zcl start`, `zcl truth build`, `zcl sync`, and `zcl doctor`.

### 2.5 Separation of Engine and Project State

- Engine code and reusable logic remain stateless and reusable.
- Project-specific runtime memory remains isolated under `.z-mos/`.
- The engine must not embed project memory.

### 2.6 Legacy Drift Elimination

- Legacy artifacts must not be reintroduced into the enforcement path.
- Existing repair flows should continue to sanitize stale references and seed current canonical paths.

## 3. Gap Analysis

This section compares the current Gen 3 Purity capabilities with the desired Gen 4 target state.

### 3.1 Architecture

| Capability | Gen 3 Purity | Gen 4 Target | Gap |
|---|---|---|---|
| Runtime topology | Single workspace / single node | Distributed multi-node / federated runtime | High |
| Engine-state separation | Strong | Preserve, but extend to federated state | Medium |
| Project memory | Local `.z-mos/` only | Local plus federated state synchronization | High |

### 3.2 Governance

| Capability | Gen 3 Purity | Gen 4 Target | Gap |
|---|---|---|---|
| Hard-block enforcement | Default fail-closed | Adaptive governance with safe policy evolution | Medium |
| Policy expressiveness | Static workflow policy + intent allowlists | Dynamic policy adaptation + trust scoring | High |
| Truth model | Static truth contract | Living, evolvable truth that remains canonical | High |

### 3.3 Intelligence

| Capability | Gen 3 Purity | Gen 4 Target | Gap |
|---|---|---|---|
| Reasoning layer | Rule-based validation | Meta-reasoning and policy-aware decision support | High |
| Agent coordination | Implicit / single agent | Multi-agent orchestration and reputation | High |
| Self-healing | Explicit repair commands only | Autonomous drift detection + repair suggestions | Medium |

### 3.4 Trace & Audit

| Capability | Gen 3 Purity | Gen 4 Target | Gap |
|---|---|---|---|
| Trace form | Append-only JSONL | Semantic trace / knowledge graph | Medium |
| Verification | Required and strict | Extend to distributed consensus verification | High |

### 3.5 Operability

| Capability | Gen 3 Purity | Gen 4 Target | Gap |
|---|---|---|---|
| Bootstrap / init | `zcl init`, `zcl memory init` | Federated bootstrap + upgrade compatibility | Medium |
| User friction | High, manual truth rebuilds | Lower friction with guided workflows | High |
| Deployment | Local project runtime | Multi-node / cloud / hybrid deployment | High |

### 3.6 Current strengths to leverage

- `sdk/runtime/run-with-intent.ts` already provides a strong enforcement gate.
- `.z-mos/truth.contract.json` and `.z-mos/intent.card.json` provide an existing authority contract model.
- `.z-mos/trace/runtime-trace.jsonl` provides an audit foundation.
- `cli/commands/init.ts` and `cli/commands/memory.ts` already include seed and repair logic.
- Documentation already defines the authority separation clearly, especially in `zmos-core.md` and `docs/user-guide.md`.

### 3.7 Key implementation risks

1. **Distributed truth synchronization** requires a new protocol and probably a new consensus or federation model.
2. **Adaptive governance** must not weaken the current hard-block guarantee.
3. **Multi-agent orchestration** introduces complexity in intent resolution, authority ownership, and trace correlation.
4. **Gen 4 naming without systemic design** risks becoming only branding, not a real architectural evolution.

This draft is intentionally grounded in the current repository baseline. It assumes the next stage of Gen 4 will evolve from the existing Purity enforcement model rather than replace it wholesale.
